CSS201 Programming 2 - BorneoFresh Market
UML Class Diagram - Complete Explanation


SECTION 1: OVERVIEW

The system has eleven classes divided into four groups.

  User Hierarchy:     User, Administrator, Customer
  Product Hierarchy:  Product, FreshProduce, OrganicProduct, DailyEssential
  Order Group:        Order, OrderItem
  Controllers:        ProductCatalogue, CustomerRegistry, BorneoFreshSystem

User and Product are abstract classes. They cannot be instantiated directly
and exist to define shared structure and behaviour for their subclasses.


SECTION 2: CLASSES


CLASS: User (abstract)

User is the base class for Administrator and Customer. It holds the fields
and methods common to both types of users.

  Fields:
    username : String
    password : String

  Methods:
    getUsername() : String
    getPassword() : String
    setUsername(username : String) : void
    setPassword(password : String) : void
    getRole() : String  [abstract]
        Must be overridden by each subclass to return the user's role.
    displayInfo() : void  [abstract]
        Must be overridden by each subclass to display relevant information.


CLASS: Administrator (extends User)

Represents a staff member who manages the product catalogue.

  Fields:

  Methods:

    getRole() : String

    displayInfo() : void

    addProduct(catalogue : ProductCatalogue, p : Product) : void

    updateProduct(catalogue : ProductCatalogue, productId : String) : void
        Updates an existing product in the catalogue.
    viewProducts(catalogue : ProductCatalogue) : void
        Displays all products, both available and unavailable.

CLASS: BorneoFreshSystem

The main entry point and controller of the application. Handles the
menu flow, authentication, and delegates to the appropriate modules.

  Fields:
    catalogue : ProductCatalogue

        A single shared Scanner instance for all console input. Only one
        Scanner on System.in should exist at a time -- closing one closes
        the underlying stream for all others.

  Methods:

        Entry point. Creates a BorneoFreshSystem instance and calls

    showMainMenu() : void

        Displays and handles the Administrator Module menu.
    showCustomerMenu(customer : Customer) : void
        Displays and handles the Customer Module menu.


SECTION 3: RELATIONSHIPS

User to Administrator and Customer (Inheritance)
  Administrator and Customer both extend User. They inherit the username
  and password fields along with their getter and setter methods.

Product to FreshProduce, OrganicProduct, and DailyEssential (Inheritance)
  All three extend Product. They inherit productId, productName, price,
  isAvailable, and their associated methods.

Customer to Order (Association, one to many)
  One customer can have zero or more orders. This is maintained via the
  orderHistory field in Customer and the customer field in Order.

Order to OrderItem (Composition, one to many)
  One order contains one or more order items. OrderItems cannot exist
  without their parent Order. If an Order is deleted, its OrderItems
  are deleted with it.

OrderItem to Product (Association, many to one)
  Many order items can reference the same product. Each OrderItem holds
  a reference to exactly one Product.

ProductCatalogue to Product (Aggregation, one to many)
  The catalogue manages a collection of products but does not own them
  in a strict lifecycle sense. A Product can conceptually exist outside
  the catalogue.

CustomerRegistry to Customer (Aggregation, one to many)
  The registry manages a collection of customers in the same way.
  A Customer can conceptually exist outside the registry.

BorneoFreshSystem to ProductCatalogue (Dependency)
  BorneoFreshSystem holds and uses a ProductCatalogue instance as one
  of its fields.

BorneoFreshSystem to CustomerRegistry (Dependency)
  BorneoFreshSystem holds and uses a CustomerRegistry instance as one
  of its fields.

Administrator to ProductCatalogue (Dependency)
  Administrator methods accept ProductCatalogue as a parameter, creating
  a usage dependency between the two classes.


SECTION 4: THE FOUR OOP PRINCIPLES

Abstraction
  User and Product are abstract classes that cannot be instantiated
  directly. They define abstract methods -- getRole() and displayInfo()
  in User, getCategory() in Product -- that subclasses are required to
  implement. The rest of the system can work with User and Product
  references without needing to know the specific subtype.

Encapsulation
  All fields across every class are private. They are accessed and
  modified only through public getter and setter methods. For example,
  the price field in Product can only be read via getPrice() and changed
  via setPrice(). This protects the internal state of each object and
  ensures changes go through controlled pathways.

Inheritance
  There are two inheritance hierarchies. In the User hierarchy,
  Administrator and Customer inherit username, password, and their
  associated methods from User. In the Product hierarchy, FreshProduce,
  OrganicProduct, and DailyEssential inherit productId, productName,
  price, isAvailable, and their methods from Product. Both avoid
  duplication and establish a clear is-a relationship between each
  subclass and its parent.

Polymorphism
  Polymorphism is achieved through method overriding. Administrator and
  Customer both override getRole() and displayInfo() from User, so
  calling getRole() on a User reference returns a different value
  depending on the actual object at runtime. Similarly, FreshProduce,
  OrganicProduct, and DailyEssential each override getCategory() from
  Product. This allows ProductCatalogue to store all product types in
  a single ArrayList<Product> and call getCategory() on any element
  without knowing its specific subtype in advance.
